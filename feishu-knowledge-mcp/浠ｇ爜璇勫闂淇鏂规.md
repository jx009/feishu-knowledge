## 修复方案总原则

这次修复不建议采用零碎补丁式处理，而是围绕**一致性闭环**做一次小范围重构，目标是：

- **飞书仍然是唯一事实源**：知识正文只以飞书为准。
- **Qdrant 只是派生索引**：可以重建，不能先于飞书存在。
- **所有知识有统一主键**：后续 `save/update/delete/sync/search/list` 都围绕同一个 `skill_id` 运转。
- **先止血再增强**：先修 P0 一致性问题，再做 chunk、维度配置、Dashboard 优化。

## 这次修复要解决的核心问题

### 一、统一身份模型

现在最大的结构问题，不是某个函数写错，而是**系统里没有一个真正稳定的“业务主键”**。

建议统一成下面这套模型：

- **`skill_id`**：唯一业务主键，贯穿全系统，不再变化
- **`feishu_doc_token`**：飞书文档实体 ID
- **`wiki_node_token`**：飞书知识库挂载节点 ID
- **`vector_point_id`**：直接等于 `skill_id`
- **`content_hash`**：当前飞书正文内容哈希，用于判断是否需要重新索引
- **`version`**：文档逻辑版本号
- **`sync_status`**：如 `PENDING_INDEX`、`INDEXED`、`PENDING_DELETE`、`FAILED`
- **`deleted`**：软删除标记
- **`last_error`**：最近一次失败原因

### 二、补一个“运行元数据注册表”

为了修好 `update/delete/sync`，建议**轻量引入一个注册表**，但它**不存正文**，只存运行元数据和映射关系。

这个注册表可以放在现有 Dashboard 的 SQLite 体系里扩展，不需要把正文落到本地，也不违背“飞书是唯一事实源”。

注册表只负责：

- **`skill_id ↔ feishu_doc_token ↔ wiki_node_token` 映射**
- 当前 `sync_status`
- `content_hash/version/updated_at`
- 重试次数和错误信息

这样以后：

- `search_skill` 走向量库
- `list_skills` / `update_skill` / `delete_skill` 走注册表 + 飞书
- `sync` 也走注册表驱动

## 分阶段修复方案

## 第一阶段：先止血，避免继续制造脏数据

### 目标

先阻止当前系统继续扩大错误状态。

### 需要做的事

- **临时下线或禁用高风险入口**
  - 暂时关闭 `vector/sync.py` 里的全量同步入口
  - 暂时限制 `tools/manage_skill.py` 里的 `update_skill` 和 `delete_skill` 对外可用性
- **禁止“飞书失败但继续写向量”**
  - `tools/save_skill.py` 的事务语义要先改设计
- **冻结 ID 扩散**
  - 明确规定：以后 Qdrant `point_id` 只能用 `skill_id`
  - 不允许再用 `node_token` 直接当 point id

### 这一阶段的验收标准

- **不会再产生新的“只有向量没有飞书”的记录**
- **不会再通过同步制造重复向量点**
- **不会再出现删除只删向量、不删飞书的情况**

## 第二阶段：统一数据模型和状态机

### 目标

给所有知识对象补一套可管理的生命周期。

### 需要调整的文件

- `knowledge/card.py`
- `dashboard/models.py`
- `dashboard/logger.py`
- `config.py`
- `config.yaml`

### 设计要点

- **`SkillCard` 增加系统字段**
  - `skill_id`
  - `feishu_doc_token`
  - `wiki_node_token`
  - `content_hash`
  - `version`
  - `sync_status`

- **向量 payload 统一**
  - 必须包含 `skill_id`
  - 必须包含 `category/project/tags/title`
  - 必须包含 `feishu_doc_token/wiki_node_token`
  - 必须包含 `content_hash/version/updated_at`
  - 必须包含 `deleted/status`

- **日志补全**
  - Dashboard 日志必须记录 `skill_id`
  - 保存 / 更新 / 删除 / 同步都要能串起来追踪

### 状态机建议

- **`CREATED_FEISHU`**：飞书已创建，还没建索引
- **`INDEXED`**：飞书和向量都一致
- **`PENDING_REINDEX`**：飞书有更新，向量待刷新
- **`PENDING_DELETE`**：飞书已删/已归档，向量待删
- **`FAILED`**：最近一次动作失败，等待补偿

## 第三阶段：修正飞书对象模型

### 目标

彻底分清**文档实体**和**知识库挂载节点**，修正当前 token 语义混乱问题。

### 需要重点改的文件

- `feishu/document.py`
- `feishu/wiki.py`
- `feishu/client.py`

### 建议拆成明确的 4 个动作

- **创建飞书文档**
  - 返回 `feishu_doc_token`
- **把文档挂到知识库指定父节点**
  - 输入 `space_id + parent_wiki_node_token + feishu_doc_token`
  - 返回 `wiki_node_token`
- **更新文档正文**
  - 输入 `feishu_doc_token`
- **删除/归档知识**
  - 明确到底是删文档、删 wiki 节点、还是仅归档
  - 建议先支持“受控归档/移除挂载”，不要直接做物理删除

### 关键原则

- **绝不能把 `wiki_node_token` 当作文档 token 用**
- **绝不能把 `category_nodes` 配置里的 wiki 节点，当作文档 API 的 folder token 用**
- **创建文档和挂载知识库必须是两个显式步骤**

### 这一阶段的验收标准

新建一条知识时，能够明确拿到：

- `skill_id`
- `feishu_doc_token`
- `wiki_node_token`
- `doc_url`

后续更新和删除，都能准确命中同一条飞书对象。

## 第四阶段：重写 `save_skill` 的事务语义

### 目标

把保存流程改成**“飞书优先、向量派生、失败可补偿”**。

### 需要重点改的文件

- `tools/save_skill.py`
- `vector/store.py`

### 新流程建议

```text
生成 skill_id
→ 组装 SkillCard
→ 创建飞书文档
→ 挂载到 wiki 分类节点
→ 写入注册表（状态 = CREATED_FEISHU）
→ 生成 embedding
→ upsert Qdrant（point_id = skill_id）
→ 更新注册表（状态 = INDEXED）
→ 记录日志
```

### 错误处理规则

- **飞书创建失败**
  - 直接失败返回
  - 不允许写入向量库

- **飞书创建成功，但向量写入失败**
  - 返回“已保存到飞书，但索引待补偿”
  - 注册表写 `PENDING_INDEX`
  - 后续由同步/补偿任务自动修复

### 这样做的好处

- 保证**永远不会出现“索引有、正文没有”**
- 即使 embedding 或 Qdrant 临时故障，也不会丢正文
- 更符合原始方案“飞书为事实源”的定位

## 第五阶段：重写 `update_skill` 和 `delete_skill`

### 目标

让管理操作真正闭环。

### 需要重点改的文件

- `tools/manage_skill.py`
- `feishu/document.py`
- `vector/store.py`

### `update_skill` 新规则

- **先查注册表拿到 `feishu_doc_token`**
- **先更新飞书正文**
- 更新成功后：
  - 计算新的 `content_hash`
  - `version + 1`
  - 写注册表为 `PENDING_REINDEX`
- 然后刷新 embedding 和向量
- 成功后更新为 `INDEXED`

#### 禁止的旧行为

- **只更新向量，不更新飞书**

### `delete_skill` 新规则

建议先做**软删除**，不要一开始就物理清理。

流程：

- 根据 `skill_id` 找到注册表记录
- 先对飞书执行“归档/从 wiki 移除/打 deleted 标记”
- 成功后把向量删掉
- 注册表写 `deleted=true`
- Dashboard 记录删除日志

#### 为什么先软删除

- 更安全
- 避免误删飞书正文不可恢复
- 方便后续做“恢复”功能

#### 禁止的旧行为

- **只删向量，不处理飞书**

## 第六阶段：重写同步策略

### 目标

让 `sync` 从“危险的全量导入器”变成“受控的一致性修复器”。

### 需要重点改的文件

- `vector/sync.py`
- `vector/store.py`

### 新同步策略

建议放弃当前“按 wiki node 直接全量刷一遍”的思路，改成**注册表驱动同步**：

- 以注册表中的 active skills 为主集合
- 拉取对应飞书文档最新内容
- 计算 `content_hash`
- 如果 hash 变化，使用同一个 `skill_id` 重新 upsert
- 如果文档已归档/已删除，则删索引并标记删除

### 明确两类同步

- **修复型同步**
  - 修复 `PENDING_INDEX` / `FAILED` / `PENDING_DELETE`
- **漂移检测同步**
  - 检查飞书手工编辑后是否需要重建向量

### 对“飞书手工新增文档”的建议

这里不要立刻自动纳管，先定策略：

- **默认不自动纳管**
  - 只同步系统已经登记过的知识
- **后续可加“导入模式”**
  - 明确在某个受控目录下，人工触发导入

### 为什么这么做

否则当前那种全量扫知识库的方式，很容易把：

- 手工文档
- 临时文档
- 旧测试文档
- 已经导入过的文档

重复变成向量点。

### 同步最关键的一条规则

- **同步时的 point id 永远只能是 `skill_id`**
- **绝不能再用 `node_token` 重新造一条向量记录**

## 第七阶段：修正搜索、列表和 Dashboard 口径

### 目标

把“检索”和“管理”彻底分层。

### `search_skill` 的职责

- 只负责语义检索
- 数据源仍然是 Qdrant
- 但必须：
  - 只搜 `deleted=false`
  - 只搜 `status=INDEXED`
  - 过滤字段统一用标准 payload

对应文件：

- `tools/search_skill.py`

### `list_skills` 的职责

- 不应该继续把向量库当管理视图主来源
- 应改为从**注册表**读取
- 支持：
  - `category`
  - `project`
  - `tags`
  - `status`
  - `deleted`

对应文件：

- `tools/list_skills.py`

### Dashboard 的职责

- 展示运行状态，不做知识正文来源
- 统计口径建议来自：
  - 注册表状态
  - 操作日志
  - 同步任务结果

对应文件：

- `dashboard/api.py`
- `dashboard/logger.py`

## 第八阶段：补上迁移和数据修复

### 目标

把当前已经写进去的历史数据修干净，不让旧脏数据继续影响系统。

### 建议的迁移顺序

- **先备份**
  - Qdrant collection 快照
  - SQLite 当前数据
  - 飞书知识库节点清单
- **扫描现有向量点**
  - 找出 `point_id = node_token` 的历史数据
  - 找出重复记录
- **回填注册表**
  - 根据现有飞书链接 / 文档 token / 节点信息建立映射
- **选择 canonical `skill_id`**
  - 对已有 `card.id` 生成的记录，优先保留原 `skill_id`
- **统一重建索引**
  - 用 canonical `skill_id` 全量重建 active skills 的向量
- **清理旧 point**
  - 删除旧的 `node_token` 风格点
  - 删除重复点

### 迁移完成后的结果

- 每条知识只有一个 `skill_id`
- 每条知识最多一个 active vector point
- 飞书、注册表、Qdrant 三者可对齐

## 哪些问题现在不要优先修

下面这些建议**放到一致性修完之后**再做：

- **chunking**
- **Markdown → Feishu Block 更完整转换**
- **embedding 维度从配置动态读取**
- **真正异步化 Feishu/Qdrant/OpenAI 调用**
- **Dashboard UI 美化**
- **自动知识提取**

它们都值得做，但现在不是主矛盾。

## 建议的文件级改造顺序

按风险和收益排序，建议这样推进：

1. `tools/save_skill.py`
2. `tools/manage_skill.py`
3. `feishu/document.py`
4. `feishu/wiki.py`
5. `vector/sync.py`
6. `vector/store.py`
7. `knowledge/card.py`
8. `tools/list_skills.py`
9. `dashboard/api.py` / `dashboard/logger.py`

## 验收标准

这套方案修完后，至少要满足下面 6 条：

- **保存**
  - 飞书失败时，不会写向量
- **保存补偿**
  - 飞书成功、向量失败时，知识仍可保留，后续能自动补索引
- **更新**
  - 飞书和向量最终一致，且只对应同一个 `skill_id`
- **删除**
  - 删除后不会再被搜索命中，也不会在飞书里失控残留
- **同步**
  - 手工编辑飞书后，只会更新原有向量，不会新增重复点
- **列表/监控**
  - `list_skills` 和 Dashboard 看到的状态与真实系统一致

## 建议的执行节奏

- **第 1 步**：先做“止血 + 主键/状态模型设计”
- **第 2 步**：修飞书对象模型和 `save_skill`
- **第 3 步**：修 `update_skill` / `delete_skill`
- **第 4 步**：重写同步和数据迁移
- **第 5 步**：最后再补 `list_skills`、Dashboard 和测试

## 结论

- **核心方向**：不是大重写，而是围绕 `skill_id + 注册表 + 飞书优先事务` 做一次有边界的修复
- **最关键决策**：把 Qdrant 明确降级为派生索引，彻底停止“飞书失败但继续写向量”的行为
- **最值得先落地的 3 个点**：
  - **统一 `skill_id`**
  - **修 `save_skill` 事务**
  - **重写 `sync` 的主键和触发逻辑**
